# Query Review

Owner: Architecture Team
Last Updated: 2026-05-12
Change Summary: Added query review coverage for the Community Connect V1 list, detail, cursor-backed members, feed, interaction, and community-owned resource/event paths.

## 1. Metadata

- Query name: Community Connect V1 Queries
- Owning module: `campus` with `social` read dependency for feed
- Runtime: `apps/backend`
- Linked LLD: `docs/lld/phase-1/COMMUNITY_CONNECT_SURFACE_LLD.md`
- Reviewer: Architecture Team
- Date: 2026-05-12

## 2. Purpose

- Support the Connect community tab, community detail pages, community member lists, community-scoped feed reads, community-scoped post writes, target-post interaction guards, and bounded community-owned resource/event previews without unbounded tenant scans.

## 3. Query Shape

- tables involved: `community_memberships`, `communities`, `tenant_memberships`, `users`, `posts`, `resources`, event dashboard store
- joins: community memberships to communities; community memberships to tenant memberships and users for member list; posts to author profile data in the social module
- filters: `membership_id`, `community_id`, `tenant_id`, `slug`, `post_id`, `comment_id`, `status`, `deleted_at is null`
- sort order: `joined_at asc` for memberships, `created_at desc` for feed/resources
- limit / pagination: no unbounded list endpoints; feed and member reads use cursor pagination

## 4. Expected Scale

- expected rows in table: communities low thousands per large tenant; community memberships low millions long-term; posts high millions long-term
- expected rows returned: list-my tens, detail one, members one page per cursor, feed one page
- frequency: Connect open, community detail open, community member expansion, community feed scroll, comment/reaction/repost mutations, resource/event preview load
- peak traffic assumption: community list read per app open; feed reads are the highest-volume path

## 5. Supporting Indexes

- existing indexes: `communities (tenant_id, slug)`, `community_memberships (community_id, membership_id)`, `posts (community_id, created_at desc)`, post/comment primary-key lookups for interaction authorization
- new indexes proposed: `community_memberships (membership_id, joined_at asc)`, `community_memberships (community_id, joined_at asc)`, `resources (community_id, created_at desc)` if not generated by Data Connect/index configuration
- why they are sufficient: every high-volume query starts from a selective viewer membership or community id and applies cursor limits

## 6. Safety Checks

- tenant filter present: yes for community detail and feed authorization
- deleted rows excluded: yes
- auth already enforced: yes, through backend viewer context; community feed reads/writes, target-post interactions, community resource reads/writes, and community event composition require the active community membership row
- unbounded scan avoided: yes, member and feed paths are paginated

## 7. Failure and Degradation

- what happens if query is slow: community list/detail shows retry or partial state; Chats tab remains usable
- fallback behavior: existing demo/fallback community response can remain development-only; member-list connector failures can degrade to the already-authorized viewer's member row, but post writes must fail closed if community membership is missing
- timeout policy: low backend connector timeout with safe retry affordance

## 8. Review Outcome

- approved: yes for V1 foundation with cursor-backed member reads and membership checks
- changes required: verify the proposed membership/resource pagination indexes before broad launch
- notes: avoid tenant-wide community member counts on the hot path; use denormalized counts or cached summaries after the first campus scale test
- notes: community resource previews now start from `community_id`; event previews use the current bounded event store and should move to indexed event rows before high-volume community event feeds
